# Light Keepers Expert Council Constitution

## 第一章：總則

### 第 1 條：設立目的

專家委員會（Expert Council）為 Light Keepers 災防平台之最高技術決策機構，負責：

1. 技術架構設計與審查
2. 程式碼品質把關
3. 系統安全評估
4. 使用者體驗優化
5. 發布策略規劃

### 第 2 條：運作原則

- **透明性**：所有決策過程需記錄於文件
- **專業性**：各領域由專責專家主導
- **效率性**：快速迭代、即時回饋
- **協作性**：跨領域專家協同作業

## 第三章：決策流程

### 第 5 條：技術審查程序

```
┌─────────────────────────────────────────────────────────┐
│                    技術審查流程                          │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  1. 需求分析 ──► 2. 方案設計 ──► 3. 專家審查            │
│        │              │              │                  │
│        ▼              ▼              ▼                  │
│   [領域專家]     [架構師]      [委員會表決]             │
│                                      │                  │
│                                      ▼                  │
│                    4. 實作 ──► 5. 驗證 ──► 6. 發布      │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 第 6 條：審查等級

| 等級 | 名稱 | 觸發條件 | 審查成員 |
|:----:|------|----------|----------|
| L1 | 快速審查 | 單一檔案修改、Bug 修復 | 1 位相關專家 |
| L2 | 標準審查 | 功能新增、API 變更 | 2 位以上專家 |
| L3 | 完整審查 | 架構變更、安全相關 | 全體委員會 |

### 第 7 條：表決規則

- **一般決議**：簡單多數通過
- **重大決議**：2/3 多數通過
- **安全相關**：資安長擁有一票否決權

## 第四章：工作模式

### 第 8 條：工作階段

專家委員會運作分為三個模式：

| 模式 | 代號 | 說明 |
|------|:----:|------|
| **規劃模式** | `PLANNING` | 研究需求、設計架構、撰寫計畫書 |
| **執行模式** | `EXECUTION` | 實作程式碼、建構功能 |
| **驗證模式** | `VERIFICATION` | 測試、審查、確認品質 |

### 第 9 條：文件產出

每項任務需產出以下文件：

1. **`task.md`** - 任務清單與進度追蹤
2. **`implementation_plan.md`** - 實作計畫（需用戶核可）
3. **`walkthrough.md`** - 完成摘要與驗證記錄

## 第五章：品質標準

### 第 10 條：程式碼標準

```yaml
TypeScript:
  - strict mode: enabled
  - ESLint: error-free
  - 型別覆蓋率: 100%

React:
  - 函數式組件優先
  - Hooks 規範遵循
  - 無障礙屬性完備

測試:
  - 單元測試覆蓋率: ≥80%
  - E2E 關鍵路徑: 100%
  
效能:
  - LCP: <2.5s
  - FID: <100ms
  - CLS: <0.1
```

### 第 11 條：安全標準

- OWASP Top 10 合規
- 敏感資料加密 (AES-256)
- JWT Token 輪換機制
- API 速率限制

### 第 12 條：無障礙標準

- WCAG 2.1 Level AA
- 鍵盤完整導航
- 螢幕閱讀器相容
- 色彩對比度 ≥4.5:1

## 第六章：緊急處理

### 第 13 條：緊急修復權限

遇重大安全漏洞或系統當機時：

1. **資安長** 或 **DevOps 工程師** 可啟動緊急修復
2. 修復後 24 小時內補齊審查程序
3. 產出事故報告（Incident Report）

### 第 14 條：回滾程序

```bash
# 緊急回滾指令
git revert HEAD --no-edit
git push origin main
npx cap sync  # 若涉及行動端
```

## 第七章：附則

### 第 15 條：憲法修訂

本憲法之修訂需經：
1. 專家委員會 2/3 多數同意
2. 專案擁有者（Project Owner）核可

### 第 16 條：解釋權

本憲法之最終解釋權歸專家委員會所有。
